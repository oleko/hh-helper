# hh-helper

Личный инструмент для поиска работы: тянет вакансии с hh.ru и SuperJob, оценивает
их через YandexGPT против твоей карьерной базы, каждый день собирает дайджест — и
по клику готовит резюме (заточенное под ATS) и сопроводительное письмо под
конкретную вакансию. Есть веб-интерфейс с логином, тремя списками (разбор/подходит/
архив), статистикой и отдельным инструментом «оцени вакансию по ссылке».

**Что инструмент НЕ делает и почему.** Он не отправляет отклики за тебя. Отклик —
твоё решение и твоя рука, после того как ты глазами посмотрел вакансию и, если
нужно, поправил письмо. Автоматизация массовых откликов не входит в объём этого
инструмента.

## Возможности

- **Три источника вакансий** — hh.ru, SuperJob и Хабр Карьера, каждый можно
  включить/выключить по отдельности в веб-интерфейсе (Хабр по умолчанию выключен —
  он IT-центричный; включи, если ищешь роли в диджитале/продукте). Хабру не нужны
  ключи: используется его открытый frontend-API поиска.
- **Скоринг через LLM** — 0–100 баллов, трек (два целевых профиля из твоей
  карьерной базы), red flags, рекомендация (подходит/подумай/пропусти), ключевые
  слова для HR/ATS-систем. Провайдер выбирается в «Настройках»: YandexGPT,
  GigaChat или любой **OpenAI-совместимый эндпоинт** — OpenAI, OpenRouter или
  своя локальная модель (Ollama, LM Studio, vLLM), без завязки на вендора.
- **Минус-слова** — вакансии, где в названии/описании встречается любое из
  заданных слов, сразу уходят в Архив без обращения к модели (экономит токены на
  заведомо ненужном). Настраивается на странице «Настройки».
- **Резюме и письмо под вакансию** — `resume_notes.md`, `resume_full.md` (готовый
  текст резюме), `cover_letter.txt` и `resume.docx` (одна колонка, стандартные
  заголовки Word, без таблиц/картинок — чтобы ATS вроде Huntflow/Потока не
  потеряли текст при разборе). Заголовок резюме — дословно название вакансии,
  достижения — по убыванию силы цифры.
- **Веб-интерфейс** — логин по сессии, три списка (Разбор → Подходит/Архив),
  пагинация, фильтры и сортировки, легенда, статистика (график по дням, зарплаты,
  недельная динамика), страница настроек (пороги, источники, карьерная база
  прямо из браузера), инструмент «вакансия по ссылке» — вставил URL, получил
  скоринг + резюме + письмо без ручного добавления в поиск.
- **Дневной AI-дайджест и автоматизация через cron** — `fetch` → `score` →
  `digest` → `check-liked` → `backup` каждое утро.
- **Коррекция по твоим решениям** — если ты не согласен с оценкой модели, это
  расхождение автоматически попадает в промпт следующего скоринга как пример.

## 1. Быстрый старт (локально)

```bash
git clone https://github.com/<your-username>/hh-helper.git
cd hh-helper
python3 -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

cp config.example.yaml config.yaml
cp .env.example .env
cp career_base.example.md career_base.md
```

Дальше — получить свои ключи и вписать их в `.env`:

1. **HH.ru** (обязательно) — https://dev.hh.ru/admin → «Мои приложения» → создать.
   Получишь `client_id`/`client_secret`. Это не твой личный логин, а отдельная
   сущность для API-доступа к публичному поиску вакансий.
2. **SuperJob** (опционально — можно выключить источник на странице «Настройки»,
   если не нужен) — https://api.superjob.ru/register/ → получишь секретный ключ.
   Публичный поиск не требует OAuth-авторизации работодателя.
3. **Yandex Cloud** (обязательно, для скоринга и генерации резюме) —
   https://console.cloud.yandex.ru/ → сервисный аккаунт → «Создать API-ключ»
   (роль `ai.languageModels.user` достаточно) + `folder_id` каталога (виден в
   консоли рядом с названием каталога, впиши в `config.yaml → yandex.folder_id`).

Впиши все ключи в `.env`, открой `config.yaml` и поправь `hh.user_agent` (формат
`ИмяПриложения/1.0 (твой-email)` — HH требует контактный email в User-Agent, это
их правило), `search.queries` (свои роли/должности) и остальное под себя.

Заполни `career_base.md` своими реальными данными (на основе структуры в
`career_base.example.md`) — это единственный файл, из которого модель берёт факты
для скоринга и резюме; она не выдумывает опыт и цифры, которых там нет.

### Первый прогон

```bash
python -m src.main fetch      # тянет вакансии по запросам из config.yaml
python -m src.main score      # оценивает новые через YandexGPT
python -m src.main digest     # печатает и сохраняет топ-вакансии в out/digest_YYYY-MM-DD.md
```

Посмотри дайджест, возьми `id` заинтересовавшей вакансии:

```bash
python -m src.main tailor 123456789
```

Положит в `out/123456789/`: `resume_notes.md` (что подчеркнуть/добавить, честно
укажет пробелы, если под требование вакансии в базе нет кейса), `resume_full.md`
(готовый текст резюме), `resume.docx` (тот же текст как Word-документ),
`cover_letter.txt` (черновик письма, 120–180 слов), `vacancy.txt` (текст вакансии
как её увидела модель).

### Веб-интерфейс

```bash
python -m src.main serve
```

Поднимет сервер на `0.0.0.0:8765` (порт/хост — `config.yaml → webapp`). Открой
`http://localhost:8765` — попадёшь на страницу входа (логин/пароль из
`WEBAPP_USER`/`WEBAPP_PASSWORD` в `.env`; для продакшн-деплоя не забудь ещё и
`WEBAPP_SECRET_KEY`, см. ниже). Сессия держится 30 дней.

## 2. Полноценный деплой (VPS + cron)

Для реального ежедневного использования `fetch`/`score`/`digest` должны идти по
расписанию, а веб-интерфейс — постоянно висеть в фоне. Пример для Ubuntu VPS.

**Сгенерируй секрет сессии** (обязательно для веб-интерфейса, иначе `serve` не
стартует):

```bash
python -c "import secrets; print(secrets.token_hex(32))"
# впиши результат в .env → WEBAPP_SECRET_KEY
```

**systemd-юнит** (`/etc/systemd/system/hh-helper-web.service`):

```ini
[Unit]
Description=hh-helper web interface
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/hh-helper
ExecStart=/opt/hh-helper/.venv/bin/python -m src.main serve
Restart=on-failure
User=root

[Install]
WantedBy=multi-user.target
```

```bash
systemctl daemon-reload
systemctl enable --now hh-helper-web
systemctl status hh-helper-web
```

**cron** (например, ежедневно в 9:00 — `crontab -e`):

```cron
0 9 * * * cd /opt/hh-helper && .venv/bin/python -m src.main fetch >> logs/cron.log 2>&1
5 9 * * * cd /opt/hh-helper && .venv/bin/python -m src.main score >> logs/cron.log 2>&1
10 9 * * * cd /opt/hh-helper && .venv/bin/python -m src.main digest >> logs/cron.log 2>&1
12 9 * * * cd /opt/hh-helper && .venv/bin/python -m src.main check-liked >> logs/cron.log 2>&1
40 9 * * * cd /opt/hh-helper && .venv/bin/python -m src.main backup >> logs/cron.log 2>&1
```

- `check-liked` — проверяет актуальность вакансий с решением 👍 «подходит» (не все
  подряд — чтобы не долбить API запросами по сотням вакансий каждый день).
- `backup` — резервная копия `hh_helper.db` в `backups/` через штатный SQLite
  `.backup()` (безопасно даже при параллельной записи); сколько копий хранить —
  настраивается на странице «Настройки» веб-интерфейса (по умолчанию 7).
- Порог автоотсечения по score (по умолчанию 40 — вакансии с оценкой не выше
  сразу уходят в архив без клика) — тоже в «Настройках», не в `config.yaml`.

Обновление кода после `git pull` — `systemctl restart hh-helper-web` (для веба);
cron сам подхватит новый код на следующий прогон.

**Важно про безопасность.** Если сервер слушает напрямую по IP без TLS — форма
логина и кука сессии поверх голого HTTP не шифруют пароль и данные при передаче.
Для личного инструмента с длинным случайным паролем риск невысокий, но если
хочешь закрыть его: ограничь входящий порт файрволом (`ufw allow from <твой IP>
to any port 8765`) до своего IP, либо поставь Caddy/nginx с автоматическим TLS
перед портом.

## 3. Деплой через Docker (альтернатива systemd+cron)

Тот же результат, что в разделе 2 (веб-интерфейс постоянно в фоне + расписание
`fetch`/`score`/`digest`/`check-liked`/`backup`), но в контейнерах — если тебе
удобнее так, чем systemd-юнит + `crontab -e`. Файлы (`Dockerfile`,
`docker-compose.yml`, `crontab`, `.dockerignore`) лежат в репозитории; ниже —
как ими пользоваться (собрать образ и проверить, что он реально стартует,
нужно самому — при подготовке этих файлов Docker не запускался).

```bash
cp config.example.yaml config.yaml   # заполнить своими данными
cp .env.example .env                 # вписать ключи
cp career_base.example.md career_base.md
touch hh_helper.db .hh_token_cache.json   # чтобы Docker не создал их как каталоги
mkdir -p out backups logs

docker compose up -d --build
docker compose logs -f web    # веб-интерфейс на :8765
docker compose logs -f cron   # supercronic — вывод job'ов видно тут же
```

Два сервиса из одного образа (`docker-compose.yml`):
- **`web`** — `python -m src.main serve`, порт 8765 наружу.
- **`cron`** — тот же образ, вместо `serve` — `supercronic /app/crontab`
  (контейнерный аналог cron: пишет в stdout, не требует демона/syslog внутри
  контейнера). Расписание — в `crontab`, по умолчанию `run-all`
  (fetch→score→digest одним вызовом) в 9:00, `check-liked` в 9:12, `backup`
  в 9:40 — поправь под себя как обычный crontab.

Оба сервиса монтируют одни и те же волюмы (`config.yaml`, `career_base.md` —
только для чтения; `hh_helper.db`, `out/`, `backups/`, `logs/`,
`.hh_token_cache.json` — на чтение/запись), поэтому видят одну и ту же базу и
не расходятся. Секреты (`.env`) читаются через `env_file`, в образ не
запекаются — `git pull` + `docker compose up -d --build` подтягивает новый код
без пересоздания волюмов.

## 4. Справочник конфига

**`config.yaml`** (не секрет, но личные настройки — не коммить в свой форк, если
он публичный):
- `search.queries` — список поисковых запросов, `search.experience` — требуемый опыт.
- `digest.min_score_to_show` — порог попадания в дайджест, `digest.top_n` —
  сколько вакансий показывать.

Город, зарплатный порог и приоритетные линии метро — не в `config.yaml`, а на
странице «Настройки» веб-интерфейса (город выбирается из готового списка, id
для hh.ru и SuperJob подставляются автоматически; для города не из списка есть
поле для ручного ввода id). Применяется сразу, без перезапуска.
- `yandex.scorer_model`/`yandex.tailor_model` — модели YandexGPT по умолчанию
  (дешёвая lite для скоринга, полная для резюме/писем) — реальный выбор модели
  и провайдера (Yandex/GigaChat/OpenAI-совместимый) теперь на странице
  «Настройки» → «Модели», список моделей берётся из `models.yaml`, а не отсюда.
- `llm.provider`/`llm.mode` — провайдер по умолчанию (пока ничего не выбрано в
  UI) и режим sync/async у Yandex (`src/llm_provider.py`, `src/yandex_client.py`).
- `gigachat.*` — опциональный второй LLM-провайдер (`src/gigachat_client.py`,
  официальная библиотека `pip install gigachat`). Убери всю секцию, если не
  нужен вообще; `credentials_env` указывает на переменную в `.env` с
  Authorization key из личного кабинета Sber Studio.
- `openai.*` — опциональный третий LLM-провайдер (`src/openai_client.py`), любой
  эндпоинт по протоколу OpenAI `/chat/completions` — не завязан на конкретного
  вендора: OpenAI, OpenRouter или **локальная модель** (Ollama, LM Studio, vLLM —
  все говорят по этому же протоколу). `base_url` — адрес эндпоинта,
  `api_key_env` — переменная в `.env` (для локальных моделей обычно не нужна
  вообще, оставь пустой). Убери секцию, если не нужен.
  **Важно (проверено вживую):** сам `api.openai.com` блокирует запросы из РФ по
  IP (`unsupported_country_region_territory`) даже с валидным ключом — с
  российского VPS напрямую не заработает, нужен OpenRouter, прокси или локальная
  модель.
- `hh.user_agent` — контактная строка для HH API (обязательна по их правилам).

**`.env`** (секреты, никогда не коммитить):
- `YANDEX_API_KEY`, `HH_CLIENT_ID`/`HH_CLIENT_SECRET`, `SUPERJOB_SECRET_KEY`,
  `GIGACHAT_CREDENTIALS`/`OPENAI_API_KEY` (оба опциональны) — ключи API сервисов.
- `WEBAPP_USER`/`WEBAPP_PASSWORD` — логин для веб-интерфейса.
- `WEBAPP_SECRET_KEY` — подписывает cookie сессии (нужен только для `serve`).

Если фильтры (регион/роль/опыт) не подходят — `python -m src.main dictionaries`
сохранит `out/areas.json`/`out/professional_roles.json`/`out/dictionaries.json`
со актуальными id от HH, сверься и поправь `config.yaml`.

## 5. Структура проекта

```
hh-helper/
  config.example.yaml     — шаблон конфига, скопировать в config.yaml
  career_base.example.md   — шаблон карьерной базы, скопировать в career_base.md
  .env.example              — шаблон секретов, скопировать в .env
  requirements.txt
  Dockerfile                 — образ для web/cron-сервисов (см. раздел 3)
  docker-compose.yml          — оба сервиса, общие волюмы
  crontab                       — расписание для supercronic внутри контейнера
  src/
    main.py                 — CLI, точка входа (fetch/score/digest/tailor/serve/...)
    hh_client.py             — обёртка над HH API
    superjob_client.py         — обёртка над SuperJob API (нормализует данные под форму HH)
    habr_client.py              — обёртка над Хабр Карьерой (frontend-API + ld+json)
    sources.py                  — диспетчер источников + разбор ссылок на вакансии
    storage.py                   — SQLite: дедуп, статусы, скоринг
    yandex_client.py               — обёртка над YandexGPT Foundation Models API
    gigachat_client.py              — обёртка над GigaChat (опциональный провайдер)
    openai_client.py                 — обёртка над любым OpenAI-совместимым эндпоинтом
    llm_provider.py                   — общий интерфейс LLMProvider поверх всех трёх
    scorer.py                          — оценка вакансии через LLM → JSON
    tailor.py                           — резюме (notes+full) + сопроводительное письмо
    docx_export.py                       — сборка resume.docx из markdown
    digest.py                             — сборка markdown-дайджеста
    webapp.py                              — веб-интерфейс (Flask)
    templates/                             — HTML-шаблоны веб-интерфейса
```

## 6. Известные ограничения (честно)

- Веб-интерфейс слушает без TLS по умолчанию (см. раздел 2) — осознанный
  компромисс ради простоты локального/VPS-деплоя, не недосмотр.
- Скоринг не видит зарплатную вилку, если работодатель её скрыл (частый случай) —
  в этом случае `salary_fit` будет «не указана».
- YandexGPT иногда своевольничает с форматом ответа при генерации резюме/письма —
  обычно помогает нажать «Перегенерировать» ещё раз.
- Ни HH, ни SuperJob, ни Хабр не отдают число откликов на вакансию через
  публичный доступ (hh.ru отдаёт `counters` только владельцу вакансии в кабинете
  работодателя, проверено вживую — по токену приложения там `null`) — поэтому
  этого показателя в статистике нет и быть не может без входа в чужой аккаунт.
- rabota.ru не интегрирован намеренно: в их публичном API (`api.rabota.ru/v6`)
  нет поиска вакансий по ключевым словам для соискателя — только получение
  вакансии по заранее известному id и эндпоинты работодателя.
- Хабр Карьера: город фильтруется по `habr_city_id` из `src/geo.py`; для городов,
  где он не заполнен, поиск идёт по всей России (рабочий фолбэк). Список
  `experience/employment/schedule`-фильтров к Хабру пока не применяется (у него
  иная модель) — только поисковая фраза и город.
